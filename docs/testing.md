# docker-view 测试方案

## 1. 目标

docker-view 的测试策略要求测试与功能同步交付。每实现一个功能，必须同时补齐前端单元测试和后端单元测试，不允许先实现功能、后补测试。

质量门槛如下：

- 前端单元测试覆盖率不低于 90%
- 后端单元测试覆盖率不低于 90%
- 新功能未达到覆盖率和关键场景覆盖要求时，不得视为完成

这里的 90% 要求既是整体目标，也是功能开发阶段的准入线。实现者需要确保新增功能不会把已有覆盖率拉低到阈值以下。

## 2. 测试分层

## 2.1 后端单元测试

后端测试重点覆盖以下层次：

- `internal/service`
  - 业务流程
  - 资源聚合
  - Docker Gateway 返回值适配
  - 错误语义归一化
  - 审计触发行为

- `internal/http`
  - 请求参数解析
  - 响应结构
  - 状态码
  - 错误码映射
  - WebSocket/SSE 入口约束

- `internal/docker`
  - SDK 适配逻辑
  - 请求参数转换
  - 错误包装
  - 流式读写边界处理

### 2.1.1 后端测试原则

- 优先用 mock 或 fake 隔离 Docker Engine 外部依赖
- 每个功能都要覆盖成功路径、失败路径和边界路径
- 对错误码、错误消息和审计触发应有显式断言
- 流式能力至少验证建链、数据透传、断开和异常收尾

## 2.2 前端单元测试

前端测试重点覆盖以下对象：

- API 客户端
  - 请求构造
  - 响应解析
  - 错误转换

- 页面组件
  - 列表加载
  - 空状态
  - 错误状态
  - 用户操作触发
  - 成功后刷新

- 领域组件
  - 容器状态展示
  - 镜像详情展示
  - 卷/网络关联关系展示
  - Compose 项目摘要
  - 日志面板
  - 终端状态提示

### 2.2.1 前端测试原则

- 优先验证用户可观察行为
- 不只测试渲染成功，要测试状态变化和异常处理
- 日志流和终端会话相关组件要验证连接中、成功、失败、关闭等状态切换
- 删除、重启、重建等高风险操作要验证确认流程和失败提示

## 3. 功能域测试要求

每个功能域都必须同时具备前端与后端单测。

### 3.1 系统概览

后端至少覆盖：

- Docker Engine 可用时统计汇总返回正确
- Docker Engine 不可用时返回统一错误

前端至少覆盖：

- 概览数据渲染
- 加载态和错误态展示

### 3.2 容器管理

后端至少覆盖：

- 容器列表过滤
- 容器详情映射
- 启动、停止、重启、删除操作
- 目标容器不存在或状态冲突时的错误处理

前端至少覆盖：

- 列表加载和筛选
- 详情页渲染
- 启停删操作的交互反馈
- 操作成功后视图刷新
- 失败提示展示

### 3.3 镜像管理

后端至少覆盖：

- 镜像列表和详情
- 镜像拉取
- 镜像删除
- 镜像不存在和删除冲突错误

前端至少覆盖：

- 镜像列表与详情显示
- 拉取镜像表单交互
- 删除确认和结果展示

### 3.4 卷管理

后端至少覆盖：

- 卷列表和详情
- 创建卷
- 删除卷
- 卷正在使用时的错误路径

前端至少覆盖：

- 卷列表与详情
- 创建和删除交互
- 关联容器展示

### 3.5 网络管理

后端至少覆盖：

- 网络列表和详情
- 创建网络
- 删除网络
- 网络正在使用时的错误路径

前端至少覆盖：

- 网络列表与详情
- 创建和删除交互
- 关联容器展示

### 3.6 Compose 管理

后端至少覆盖：

- 项目识别
- 项目列表与详情
- up、down、restart、recreate 行为编排
- 项目不存在与部分容器异常场景

前端至少覆盖：

- 项目列表与详情渲染
- 项目级操作交互
- 项目状态变化反馈

### 3.7 日志流

后端至少覆盖：

- 日志查询参数解析
- SSE 事件编码
- 日志流关闭与异常处理

前端至少覆盖：

- 历史日志展示
- 流式追加展示
- 断流和错误提示

### 3.8 终端会话

后端至少覆盖：

- exec 会话创建
- WebSocket 消息类型处理
- resize、stdin、close 行为
- 容器不可 exec 时的错误路径

前端至少覆盖：

- 会话建立和关闭状态展示
- 输入透传
- 异常断开提示
- 会话重连或重新创建入口提示

## 4. 覆盖率要求

## 4.1 覆盖率指标

建议采用如下最低标准：

- 后端 statement 覆盖率 >= 90%
- 前端 statement 覆盖率 >= 90%
- 条件允许时同时跟踪 branch 覆盖率，目标也应接近 90%

## 4.2 覆盖率执行策略

- 每次新增功能时都执行相关测试
- 合并前执行完整前端单测和后端单测
- 若覆盖率下降到 90% 以下，必须补测后再继续
- 对关键高风险模块，覆盖率高于阈值仍不代表可以缺失异常路径测试

## 5. 建议的执行命令

结合当前仓库结构，建议后续实现阶段采用以下检查方式：

### 5.1 后端

```bash
go test ./... -cover
```

### 5.2 前端

在 `web/` 目录下执行：

```bash
pnpm test
```

后续建议补充覆盖率命令，使前端测试输出覆盖率报告并纳入 CI 门禁。

## 6. 完成定义补充

一个功能只有在同时满足以下条件时才视为完成：

- 后端实现完成
- 前端实现完成
- 后端单元测试完成
- 前端单元测试完成
- 相关覆盖率达到 90% 目标
- 关键成功路径、失败路径和边界路径已验证

任何只完成代码、不完成测试的提交都不符合本项目定义的完成标准。
