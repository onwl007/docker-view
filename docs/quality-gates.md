# docker-view CI 与质量门禁

## 1. 目标

本文档定义 docker-view 在实现阶段必须执行的自动化检查项、覆盖率门槛和合并门禁，确保代码质量要求可以被持续执行，而不只是停留在设计层面。

质量门禁的核心原则如下：

- 每次提交都必须可被自动化验证
- 每个功能都必须同时具备前端和后端测试
- 覆盖率要求必须纳入 CI，而不是依赖人工检查
- lint、typecheck、unit test 是最小门禁，不可跳过

## 2. 最小门禁集合

每次功能开发完成后，至少需要通过以下检查：

### 2.1 后端

- `go test ./... -cover`
- `go test ./...`

如果后续补充静态分析，建议加入：

- `go vet ./...`

### 2.2 前端

在 `web/` 目录下执行：

- `pnpm lint`
- `pnpm typecheck`
- `pnpm test`

后续应补充覆盖率输出命令，并将其纳入硬性门禁。

## 3. 覆盖率门槛

## 3.1 基线要求

- 后端单元测试覆盖率不得低于 90%
- 前端单元测试覆盖率不得低于 90%

## 3.2 覆盖率门禁策略

- 当新增功能导致覆盖率跌破 90% 时，合并请求不得通过
- 不允许通过删减测试统计范围来规避门槛
- 不允许以“后续补测试”为理由暂时绕过覆盖率要求
- 高风险模块即使已经达到 90%，仍要覆盖失败路径和边界路径

## 4. 合并请求门禁

一次功能提交在进入主分支前，至少满足以下条件：

- 前端代码实现完成
- 后端代码实现完成
- 前端单元测试已补齐
- 后端单元测试已补齐
- 前端覆盖率达标
- 后端覆盖率达标
- lint 通过
- typecheck 通过
- 文档与实现保持一致

以下情况应直接视为不满足合并条件：

- 只完成后端实现，前端测试缺失
- 只完成前端页面，后端测试缺失
- 功能能跑通，但错误路径未覆盖
- 流式能力只有手工验证，没有自动化测试

## 5. 建议的 CI 流程

建议后续在 CI 中至少拆分为以下阶段：

### 5.1 Backend Checks

- 安装 Go 依赖
- 执行后端单元测试
- 生成覆盖率结果
- 校验覆盖率是否达到 90%

### 5.2 Frontend Checks

- 安装 `web/` 依赖
- 执行 `pnpm lint`
- 执行 `pnpm typecheck`
- 执行前端单元测试
- 生成覆盖率结果
- 校验覆盖率是否达到 90%

### 5.3 Optional Integration Checks

在实现阶段成熟后可以增加：

- API 集成测试
- Docker Engine 联调测试
- 流式日志和终端的端到端验证

这些检查可以作为增强项，但不能替代前后端单元测试门禁。

## 6. 功能完成定义

功能完成必须同时满足以下条件：

- 功能代码完成
- 前端测试完成
- 后端测试完成
- 前端覆盖率不低于 90%
- 后端覆盖率不低于 90%
- 相关 lint 和 typecheck 通过
- 文档同步更新

只要缺少其中任一项，就不能认定为“完成”。

## 7. 例外处理原则

原则上不允许例外。

如果未来确实存在短期无法测试的特殊场景，必须满足以下全部条件：

- 明确说明未覆盖原因
- 明确说明风险范围
- 明确说明补测计划和截止时间
- 经代码审查显式批准

即便如此，也不应影响 90% 总体覆盖率门槛。
