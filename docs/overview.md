# docker-view 整体设计总览

## 1. 文档目标

本文档用于定义 docker-view 的产品目标、能力范围、设计边界和首版实现原则，作为后续架构设计、接口设计和开发实施的上位说明。

docker-view 的目标是提供一个运行在浏览器中的容器管理产品，聚焦 Linux 主机上的 Docker 容器资源管理，提供接近桌面容器管理工具的使用体验，但不引入 Kubernetes、Swarm 或其他集群编排能力。

## 2. 产品定位

docker-view 是一个单体部署的 Web 应用，由 Go 后端和 React 前端组成，后端通过 Go 的 Docker API 客户端直接调用 Docker Engine。

首版的定位是：

- 面向单台 Linux 主机的容器管理
- 支持从 Web 端查看和管理容器相关核心资源
- 支持进入容器执行命令
- 支持查看和持续跟踪容器日志
- 支持管理 Docker Compose 项目

不在首版范围内的能力：

- Kubernetes
- Docker Swarm
- 多主机统一纳管
- 镜像仓库平台能力
- 主机级 shell 远程登录
- 非 Docker 容器引擎适配

## 3. 首版范围

首版必须覆盖以下资源域和操作域。

### 3.1 概览

- 主机基础信息
- Docker Engine 连接状态
- 容器、镜像、卷、网络统计信息
- Compose 项目数量和运行状态概览
- 最近操作与关键告警预留位

### 3.2 容器管理

- 查看容器列表和详情
- 按名称、状态、标签过滤
- 启动、停止、重启、删除容器
- 查看容器挂载、端口映射、网络信息、环境变量摘要
- 查看容器日志
- 进入容器执行命令

### 3.3 镜像管理

- 查看镜像列表和详情
- 按仓库名、标签过滤
- 拉取镜像
- 删除镜像
- 查看镜像关联容器关系

### 3.4 卷管理

- 查看卷列表和详情
- 创建卷
- 删除未使用卷
- 查看卷被哪些容器使用

### 3.5 网络管理

- 查看网络列表和详情
- 创建网络
- 删除未使用网络
- 查看网络关联容器

### 3.6 Compose 管理

- 查看 Compose 项目列表
- 查看项目详情和关联容器
- 启动项目
- 停止项目
- 重建项目
- 删除项目及关联资源的策略定义

### 3.7 实时能力

- 容器终端：通过 WebSocket 建立双向交互会话
- 容器日志 tail：通过 SSE 推送单向实时日志流

## 4. 用户场景

docker-view 首版主要服务以下高频场景：

- 开发或运维人员查看当前 Docker 主机的资源状态
- 通过浏览器定位容器异常、查看日志和状态
- 对单个容器执行常见生命周期操作
- 进入容器执行诊断命令
- 对 Compose 项目做启动、停止和重建操作
- 查看卷、网络与容器之间的关系，辅助排障

## 5. 核心设计原则

### 5.1 单机优先

首版后端进程部署在目标 Linux 主机上，直接访问本机 Docker Engine。系统设计上保留多主机扩展边界，但不引入多主机连接、注册、调度和权限模型。

### 5.2 直连 Docker API

后端统一使用 Go 的 Docker API 客户端访问 Docker Engine，不通过 shell 调用 `docker` 命令实现核心业务，避免命令拼接、行为漂移和额外安全风险。

### 5.3 REST 为主，实时通道按需补充

- 资源查询和管理操作以 REST 为主
- 容器终端使用 WebSocket
- 日志流使用 SSE

### 5.4 受控环境优先

首版默认部署在受控内网环境，不实现完整用户系统，但必须为后续认证、授权和审计扩展预留明确接入点。

### 5.5 渐进式交付

实现顺序遵循先只读、再写操作、最后实时能力和 Compose 能力的策略，降低初期复杂度并提升可验证性。

## 6. 非功能要求

### 6.1 可维护性

- 后端遵循清晰分层
- Docker Engine 访问通过显式接口隔离
- HTTP Handler 保持轻量
- 前端采用 React + Vite 单应用结构，保持页面域清晰，避免在组件渲染层堆积业务逻辑
- 前端路由、服务端状态和展示组件职责分离，分别由 TanStack Router、TanStack Query 和基于 Tailwind CSS / shadcn/ui 的组件层承载

### 6.2 可扩展性

- 资源模型和 API 版本从首版开始固定在 `/api/v1`
- 架构上预留未来多主机接入层
- 预留认证授权中间件和审计上下文字段

### 6.3 安全性

- 不暴露主机原始 shell
- 所有容器内命令执行能力都必须限定在指定容器上下文
- 所有写操作和敏感读操作都应支持审计
- 避免在配置文件中引入明文敏感信息

### 6.4 测试质量

- 每实现一个功能，都必须同时补齐前端单元测试和后端单元测试
- 前端和后端都应建立可持续执行的自动化测试基线
- 项目整体测试覆盖率目标不低于 90%
- 新增功能在合入前应证明其相关代码路径已被测试覆盖
- 缺少测试的功能视为未完成
