# docker-view 实施路线图

## 1. 路线原则

实施顺序遵循以下原则：

- 先打通基础架构和只读能力
- 再补充资源写操作
- 再实现实时日志和容器终端
- 最后补强 Compose 管理和安全能力
- 每个阶段都要求对应前后端单元测试同步落地
- 每个阶段结束时，前后端覆盖率目标都不低于 90%

这样可以保证每个阶段都可演示、可测试、可回归。

## 2. Phase 1: 基础骨架与系统概览

目标：

- 完成后端服务基础路由结构
- 接入 Docker Gateway 最小实现
- 完成前端基础布局和路由
- 提供主机和 Docker Engine 概览接口

完成条件：

- 前端能展示系统概览页
- 后端能返回 Docker Engine 健康与基础统计
- API 返回结构和错误结构稳定
- 概览功能具备前端和后端单元测试

## 3. Phase 2: 核心资源只读能力

目标：

- 完成容器、镜像、卷、网络的列表与详情接口
- 完成前端对应页面
- 支持基础过滤与详情查看

完成条件：

- 浏览器可完成四类资源浏览
- 详情页能展示关联关系
- 只读 API 具备基础单元测试
- 前端页面和 API 客户端具备对应单元测试
- 当前累计覆盖率维持在 90% 以上

## 4. Phase 3: 资源管理操作

目标：

- 支持容器启动、停止、重启、删除
- 支持镜像拉取、删除
- 支持卷和网络创建、删除
- 接入审计事件记录

完成条件：

- 所有写操作都能返回稳定结果和错误语义
- 写操作都已接入审计
- 前端交互具备确认和错误提示
- 每个操作都具备前后端单元测试
- 当前累计覆盖率维持在 90% 以上

## 5. Phase 4: 日志与终端

目标：

- 实现容器日志查询接口
- 实现日志 SSE 实时流
- 实现 exec 会话创建接口
- 实现终端 WebSocket 双向交互

完成条件：

- 可在浏览器实时查看日志
- 可在浏览器进入容器执行命令
- 异常断开和容器退出场景处理明确
- 流式能力具备前后端单元测试
- 当前累计覆盖率维持在 90% 以上

## 6. Phase 5: Compose 管理

目标：

- 实现 Compose 项目列表和详情
- 实现项目级启动、停止、重建操作
- 在前端展示项目和容器关联关系

完成条件：

- 浏览器可完成基础 Compose 项目管理
- Compose 项目状态和容器状态映射规则明确
- 错误场景有统一处理
- Compose 能力具备前后端单元测试
- 当前累计覆盖率维持在 90% 以上

## 7. Phase 6: 安全与可运维性增强

目标：

- 补基础认证方案
- 完善审计落盘
- 增强错误观测与运行日志
- 补部署文档和运维约束

完成条件：

- 文档中定义的安全基线具备实现落点
- 审计事件可查询或可导出
- 部署约束和风险提示齐全
- 覆盖率门槛和测试门禁纳入持续集成

## 8. 验收建议

每个阶段结束都至少验证以下内容：

- 后端单元测试
- 前端关键页面行为测试
- 前端和后端覆盖率不低于 90%
- 关键 API 手工联调
- 错误场景回归
- 文档与实现一致性检查
- CI 门禁规则与本阶段实现保持一致
