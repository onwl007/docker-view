# docker-view 安全设计

## 1. 安全基线

docker-view 首版默认部署在受控内网环境，但这不是放松安全边界的理由。由于后端服务直接操作 Docker Engine，系统天然具备高权限能力，因此必须显式约束攻击面。

首版安全基线如下：

- 浏览器不直接访问 Docker Engine
- 所有 Docker Engine 调用只经过后端服务
- 不提供主机级 shell 入口
- 容器内命令执行必须绑定明确的容器上下文
- 所有写操作和敏感会话操作都应可审计

## 2. 主要风险面

### 2.1 Docker Engine 高权限风险

一旦后端服务可访问 Docker Socket，就相当于具备较高主机控制能力。系统必须避免把这种能力无约束暴露给前端。

约束要求：

- 后端只能开放受控 API，而不是代理任意 Docker API
- 不允许前端传入任意 Docker 请求路径
- 不允许把 Docker SDK 对象原样暴露到前端

### 2.2 容器终端风险

容器 exec 是高风险能力，可能被用于查看密钥、修改容器状态或横向探索。

约束要求：

- 终端仅允许附着到指定容器
- 必须记录会话创建、关闭和异常中断事件
- 应记录执行命令、目标容器和来源标识
- 文档中明确：该能力不等价于主机 shell

### 2.3 日志泄露风险

容器日志可能包含敏感信息。

约束要求：

- 日志接口不得暴露主机文件路径
- 应限制返回内容为容器日志语义
- 未来引入认证后，日志查看权限应与写操作权限解耦设计

## 3. 首版安全策略

## 3.1 网络暴露

- 默认只对受控网络开放服务
- 不建议直接暴露到公网
- 若需要公网访问，应放在反向代理与额外认证之后

## 3.2 认证与授权

首版不实现完整用户体系，但文档必须预留以下接入点：

- HTTP 认证中间件
- 请求上下文中的 actor 信息
- 审计事件中的 actor、source IP、user agent
- 写操作统一鉴权钩子

当前实现状态：

- 已实现首版静态 token 鉴权接入点
- 当前通过 `security.requireAuthentication` 和 `security.authToken` 控制敏感 API 访问
- 当前更适合作为内网或外层网关后的最小鉴权补丁，不是完整用户系统

在未接入认证前，系统的部署前提是：

- 仅在可信网络内使用
- 由外层网关或内网隔离控制访问范围

## 3.3 审计

以下行为必须具备审计能力：

- 容器生命周期操作
- 镜像拉取和删除
- 卷和网络创建删除
- Compose 项目操作
- 终端会话创建与关闭

审计事件建议包含：

- `eventType`
- `targetType`
- `targetId`
- `action`
- `actor`
- `source`
- `result`
- `timestamp`
- `details`

当前实现状态：

- 已实现内存态审计存储
- 已提供最近审计事件查询和 NDJSON 导出
- 当前未实现持久化落盘和独立查询权限模型

## 4. 配置与密钥

- 避免在配置文件中保存明文敏感信息
- 环境变量和配置文件都应视为可审计输入
- 若未来支持远程 Docker API 或 TLS，应把证书路径作为配置项，而不是内联内容

## 5. API 安全约束

- 所有输入参数都必须做显式校验
- 资源 ID、名称、过滤条件都应限制长度和格式
- 不能允许通过 API 透传任意命令到主机环境
- 不能暴露用于执行任意主机命令的接口

## 6. 实时通道安全约束

### 6.1 WebSocket 终端

- 会话建立前必须验证目标容器存在且状态允许
- 每个终端会话都应有唯一会话标识
- 断开连接后应清理相关资源
- 未来引入认证后，WebSocket 握手必须继承身份上下文

### 6.2 SSE 日志

- SSE 只用于日志流
- 不应复用为通用事件总线
- 异常断线后的重连策略必须可控，避免无上限重试

## 7. 后续增强方向

安全增强的后续优先级建议如下：

1. 增加基础登录和会话管理
2. 为写操作增加最小角色模型
3. 为 exec 和日志查看增加更细的权限边界
4. 增加审计持久化与查询能力
5. 在反向代理层引入 TLS、来源限制和速率限制
